Wisp is a free, self-hostable, drop-in tool for sites where an account is a convenience, not an obligation. The browser passes a quick humanity check, then the user picks how they'll come back — a passkey, a copyable code, or nothing at all.

How it works

Two steps, then the trail goes cold

No personal information at any point — and less kept behind at every step.

STEP 01

Pass the bot filter

The browser solves a small proof-of-work puzzle — a bot filter that costs a moment of CPU, not an email address or a CAPTCHA farm. No IP logging, no fingerprinting — that covers Wisp itself; the web server in front of it keeps its own access logs.

Under the hood: the server hands your browser a sealed, single-use challenge, and the browser grinds through ~33,000 hashes — about a second — to find a winning answer the server can check instantly. It's a filter, not a wall: it screens out off-the-shelf spam bots and makes bulk abuse cost real electricity, which is exactly why the label below only ever claims "probably human."

STEP 02

Pick how you return

A passkey (Face ID / Windows Hello), an account code you save yourself, or a guest session with nothing that links back to you. Operators choose which modes to offer.

RESULT

The server stores almost nothing

A random user ID, a timestamp, and — depending on mode — a public key or a hash of your code. There is no email to recover with, and nothing that ties the account to who you are.

The honesty contract

What an account here actually proves

The honesty contract is the product. Wisp returns exactly three confidence labels and never claims more — no "one account per human", ever.

bot-filtered

Probably human. Passed the proof-of-work check. That's the whole claim — nothing about identity.

same-passkey return

Same passkey as before. Presented a known passkey (it may sync across your devices). Usually the same person.

same-code return

Whoever holds the code. Presented a valid account code — custody of the code is the only claim.

Choosing a mode

Three ways to come back

Each with an honest trade-off — pick the one that fits how you'll return.

  Passkey Account code Guest
How you return Face ID, Windows Hello, or a fingerprint on this device Paste the code you saved You don't — it's ephemeral
Works across devices Only if your passkeys sync (iCloud / Google Password Manager) Yes — anywhere you can paste the code No

Pick a passkey if you'll mostly return on your own device and want one-tap sign-in. Pick a code if you need to get back in from anywhere and are happy to store a secret yourself. Pick guest if you just want to try the site.

Add it to your site

Drop in one script tag

Run it locally, drop in one script tag, then use the prebuilt buttons — or call the API from your own.

01 Run it locally

Builds the widget and starts the Hono server. This page is served at the root.

npm install
npm run dev
# → http://localhost:8787

02 Drop in the widget

One script tag exposes a global Wisp. Wire it to any button you like.

<script src="/wisp.js"></script>
<script>
  // same origin by default; point elsewhere if needed:
  Wisp.configure({ baseUrl: "https://accounts.example.com" });

  const account = await Wisp.signUp();
  // { userId: "…", confidence: "bot-filtered" }
</script>

05 Self-host configuration

All server config is environment variables. Set WISP_SECRET in production, or sessions reset on every restart. Serve over HTTPS in production — account codes and session cookies travel over the wire, and only HTTPS keeps them from being read in transit.

Good to know

The honest fine print

There is no email to recover with. A passkey account lives on your device — keep it backed up (iCloud / Google Password Manager) or losing the device means starting fresh. A code account lives wherever you saved the code — anyone holding the code holds the account. A guest account lasts only as long as this browser keeps its cookie.
Scanning a QR code with your phone to make a passkey? Your phone picks the passkey app, not this site. Microsoft Authenticator only stores Microsoft-account passkeys and will refuse with "doesn't support this kind of key" — choose your phone's built-in password manager (Apple Passwords / Google Password Manager) or a manager like 1Password or Bitwarden instead, under the phone's AutoFill settings.
